# 2. 玩法与关卡设计文档 (Level Design Spec)

## 2.1 文档目标

本文档用于定义横版射击关卡的结构规范、生产流程和验收标准，保证以下内容可协同推进：

- `Tilemap` 地图搭建
- `MapData / PlatformDef` 配置填写
- `MonsterSpawner` 波次配置
- 主线章节节奏设计
- 场景联调与测试回归

本项目的关卡设计必须同时满足三类目标：

- 给玩家提供持续的移动、射击和技能决策
- 为数值和 AI 提供稳定的验证场景
- 能够被当前工程的 Tilemap 导出与副本流程稳定消费

## 2.2 关卡设计原则

- 横版可读性优先于纯视觉复杂度
- 每一段平台结构都应服务于战斗、移动或信息传达
- 关卡难度优先通过敌人组合、火力线和空间限制提升
- 新机制首次出现必须提供低风险学习空间
- 关卡节点必须能映射到 `MainChapter`、`MapData`、`MonsterSpawner` 的配置链路

## 2.3 关卡结构分层

### 2.3.1 地图层级

基于当前工程的 Tilemap 生产方式，地图建议拆为以下层：

- `Collision`：逻辑阻挡层，参与碰撞导出
- `Platform`：单向平台或可跳落平台
- `Visual_Back`：背景与远景
- `Visual_Mid`：主视觉地表
- `Visual_Front`：前景遮挡与装饰
- `Trigger`：区域触发、相机切换、波次启动、剧情节点

规范：

- 阻挡层必须只包含逻辑需要的碰撞信息
- 视觉层不得混入逻辑碰撞 Tile
- `Trigger` 区域需要命名，并与章节设计稿一一对应

### 2.3.2 空间模块

建议将关卡拆成以下标准空间模块，便于复用：

- `教学段`：单一机制、低风险、低怪密度
- `移动段`：跳台、落差、移动平台
- `遭遇段`：1 到 2 波常规怪组合
- `封锁段`：关闭出口，必须清怪或触发机关
- `恢复段`：低压区、掉落、补给、信息提示
- `Boss 预热段`：逐步提示 Boss 核心机制
- `Boss 段`：多阶段战斗和专属演出

## 2.4 地图元素规范

### 2.4.1 平台与落差

建议首版遵循以下规则：

- 单个基础跳跃间距不超过玩家最大跳跃距离的 `70%`
- 关键路径不得要求连续极限跳操作
- 战斗段上方平台应允许玩家进行垂直位移规避
- 掉落后应有回归主路径的补救路线，除非该区域设计为惩罚区

### 2.4.2 常用机关

首版机关优先做“简单、明确、好调试”的类型：

- 单向门：控制推进方向
- 激光陷阱：制造时机压力
- 移动平台：服务平台跳跃和火力规避
- 电流地板：区域封锁与站位惩罚
- 传送点：切换波次或小场景

机关要求：

- 机关触发条件必须可视化
- 机关伤害应使用统一伤害来源，便于统计与调试
- 机关不得在玩家第一次见到时同时叠加高压怪群

## 2.5 波次与刷怪设计

### 2.5.1 触发方式

当前工程可按以下方式组织波次：

- 进入触发区后启动
- 击杀上一波全部敌人后启动
- 到达指定进度后启动
- 与机关、剧情、Boss 阶段联动启动

每一波怪建议记录以下设计信息：

- 波次编号
- 触发条件
- 怪物组合
- 刷新位置
- 结束条件
- 奖励或掉落
- 是否封锁出口

### 2.5.2 怪物组合模板

首版常用组合建议如下：

| 组合模板 | 内容 | 设计目的 |
| :--- | :--- | :--- |
| 压迫前线 | 近战 2 + 远程 1 | 教玩家边打边移 |
| 双向包夹 | 左右各刷 1 组 | 测试转向与撤退 |
| 高低差火力 | 地面近战 + 高台射手 | 迫使垂直位移 |
| 坦克护射手 | 坦克 1 + 远程 2 | 强化目标优先级 |
| 高速骚扰 | 高速近战 2 + 飞行 1 | 打乱玩家节奏 |

规则：

- 同屏首次出现的新敌人类型不超过 2 种
- 普通波次不建议同时加入过多控制型机关
- 精英波次应有明显识别点和奖励预期

## 2.6 节奏设计

### 2.6.1 单关标准节奏模板

建议单个主线关卡时长控制在 `8 - 15 分钟`，节奏可按以下结构组织：

1. 开场引导 `30s - 60s`
2. 基础战斗教学 `1 - 2 分钟`
3. 第一轮稳定推进 `2 - 3 分钟`
4. 中段高压遭遇 `2 - 4 分钟`
5. 过渡与补给 `30s - 60s`
6. Boss 或结尾封锁战 `2 - 4 分钟`
7. 结算与离场 `30s`

### 2.6.2 节奏控制手段

关卡设计可通过以下手段调节节奏：

- 怪物刷新密度
- 垂直空间复杂度
- 机关开关频率
- 补给点稀缺程度
- 封锁战持续时间
- 相机压迫感和视野开阔度

## 2.7 主线章节建议

### 2.7.1 第一章：入门与基础操作

目标：

- 教会玩家移动、跳跃、普通攻击
- 让玩家认识 2 种普通敌人和 1 类陷阱

配置建议：

- 低压地形
- 小规模怪群
- 至少 1 段安全平台跳跃
- 不安排长时间封锁战

### 2.7.2 第二章：多方向压力

目标：

- 引入远程压制和高低差火力
- 让玩家学会使用技能拆场

配置建议：

- 高台射手
- 2 段封锁战
- 至少 1 个移动平台或激光区

### 2.7.3 第三章：机制整合与 Boss

目标：

- 综合考查资源管理、技能使用和走位判断
- 形成完整的主线里程碑体验

配置建议：

- 普通怪、精英怪、机关混编
- 明确的 Boss 两阶段或三阶段节奏
- 允许玩家在战前识别 Boss 机制提示

## 2.8 配置落地规范

### 2.8.1 与配置表对应关系

建议使用以下职责分工：

- `TbMapData`：地图资源入口、逻辑地图标识、基础场景信息
- `TbPlatformDef`：平台、阻挡、碰撞相关定义
- `TbMainChapter`：章节与关卡入口
- `TbMonsterSpawner`：刷怪点、波次、触发条件

配置流程：

1. 在 Tilemap 搭建关卡
2. 导出碰撞数据
3. 写 `MapData` / `PlatformDef`
4. 在 `MainChapter` 绑定地图
5. 在 `MonsterSpawner` 配置刷怪
6. 进游戏验证视觉地图与逻辑地图一致

### 2.8.2 命名规范

- 地图场景：`ChapterXX_MapYY`
- 刷怪点：`WaveXX_PointYY`
- 触发器：`Trigger_功能_序号`
- Boss 段标记：`BossPhase01 / BossPhase02`

## 2.9 奖励与补给布置

奖励投放必须服务于节奏，而不是纯随机堆点：

- 教学段后投放少量恢复物，强化正反馈
- 高压波次后投放奖励或补给，缓冲挫败感
- Boss 前应提供一次明确整理资源的机会
- 隐藏房间或支线平台可放额外掉落，鼓励探索

禁止：

- 在主线必经路径连续投放过多恢复品
- 依赖掉落数量掩盖关卡难度过高问题
- 把关卡奖励全部堆在最终结算，导致中途正反馈不足

## 2.10 关卡验收标准

每个关卡至少通过以下检查：

- 从入口到结算可完整跑通
- 主路径无卡死、无不可回收掉落、无必现穿模
- 逻辑碰撞与视觉碰撞一致
- 刷怪触发条件稳定，不重复触发或漏触发
- 玩家死亡、重进、断点重试后状态可恢复
- 关卡平均完成时长落在目标区间
- 首次体验时能明确理解核心机制

## 2.11 关卡评审模板

每次新关卡评审建议按以下格式输出：

- 关卡目标
- 新增机制
- 核心敌人组合
- 节奏曲线
- 关键风险点
- 需要验证的系统
- 配置表清单
- 资源与特效需求
- QA 重点回归项

## 2.12 当前关卡配置字段落地说明

以下字段基于当前项目的真实 `cfg` 结构，适合直接给策划和程序对表使用。

### 2.12.1 `MainChapter`

当前字段非常精简：

| 字段 | 含义 | 当前作用 |
| :--- | :--- | :--- |
| `Id` | 章节/关卡编号 | `FightProxy -> MainDungeonProxy` 的入口 ID |
| `MapId` | 地图编号 | 用于加载 `MapData` |

说明：

- 当前 `MainChapter` 只承担“关卡入口到地图”的映射
- 章节名、推荐等级、奖励预览、解锁条件等信息如果后续要做，需要扩字段或补表

### 2.12.2 `MapData`

当前字段：

| 字段 | 含义 | 生效位置 | 常见错误 |
| :--- | :--- | :--- | :--- |
| `Id` | 地图编号 | `MapProxy.LoadMap` | 与 `MainChapter.MapId` 不一致 |
| `Name` | 地图资源名 | 场景碰撞与渲染加载 | 资源名和实际场景不一致 |
| `Random` | 是否随机地图 | 预留 | 当前若未用可先固定 |
| `Birth` | 出生点 | 玩家初始出生位置 | 出生点落在阻挡内 |
| `Music` | 地图音乐 | 场景音频 | 音乐资源缺失 |
| `PlatformDataId` | 动态平台列表 | `MapProxy.CreatePlatform` | 平台 ID 配错或漏配 |

### 2.12.3 `PlatformDef`

当前字段：

| 字段 | 含义 | 生效位置 |
| :--- | :--- | :--- |
| `Id` | 平台编号 | `MapData.PlatformDataId` 引用 |
| `StartX / StartY` | 起点 | 平台逻辑初始位置 |
| `EndX / EndY` | 终点 | 平台往返目标点 |
| `Speed` | 移动速度 | 平台移动 |
| `HalfWidth / HalfHeight` | 半宽半高 | 站立判定 |
| `AssetPath` | 资源路径 | 渲染层资源加载 |

常见错误：

- `HalfWidth / HalfHeight` 按美术尺寸直接填整宽整高，导致站立判定偏差
- 起终点过近但速度过大，造成平台抖动观感
- `AssetPath` 可渲染但逻辑体积与视觉尺寸不一致

### 2.12.4 `MonsterSpawner`

当前字段：

| 字段 | 含义 | 生效位置 |
| :--- | :--- | :--- |
| `Id` | 刷怪点编号 | 运行时状态索引 |
| `MonsterId` | 怪物配置 ID | 创建怪物 |
| `TriggerX` | 玩家 X 超过后触发 | `MonsterSpawnerProxy.LogicUpdate` |
| `SpawnX / SpawnY` | 刷新绝对坐标 | 怪物出生位置 |
| `PatrolMode` | 巡逻坐标模式 | 0 相对生成点，1 绝对坐标 |
| `PatrolCenterX` | 巡逻中心 X | 怪物巡逻中心 |
| `PatrolHalfRange` | 巡逻半径 | 怪物巡逻范围 |
| `MaxCount` | 生成数量 | 一次性生成数量 |

当前限制：

- 每个触发点当前只触发一次
- 触发条件目前是“任意玩家 X 大于 TriggerX”
- 还没有原生波次组、连锁触发、关门清怪等更复杂结构

这意味着：

- 现阶段关卡设计更适合线性推进式遭遇
- 如果要做完整波次战，建议后续扩 `MonsterSpawner` 或加 `WaveDef`

## 2.13 当前三章可执行拆解

### 2.13.1 第一章：基础操作验证关

目标：

- 验证登录到进图闭环
- 验证角色移动、跳跃、普通射击、基础刷怪

配置任务：

- 建一个 `MainChapter`
- 配一个 `MapData`
- 至少 1 个玩家出生点
- 至少 2 个 `MonsterSpawner`
- 至少 2 种普通敌人

程序任务：

- 跑通 `FightMainChapter -> FightProxy.EnterFight`
- 确认 `MapProxy.LoadMap`
- 确认 `UnitProxy.InitRole`
- 确认 `MonsterSpawnerProxy` 触发正常

美术任务：

- 基础地图块
- 角色基础动画
- 普通子弹特效
- 小怪受击和死亡表现

测试任务：

- 首次进入关卡
- 重复进入关卡
- 玩家出生点正确
- 怪物触发点正确

### 2.13.2 第二章：高低差和技能拆场验证关

目标：

- 验证高低差地图
- 验证远程怪、移动平台、技能和 AOE

配置任务：

- 新增包含高台的 `MapData`
- 至少 1 组 `PlatformDef`
- 至少 1 个远程怪配置
- 至少 1 个 AOE 技能配置

程序任务：

- 验证平台携带角色移动
- 验证 `Timeline -> UseAoeToPoint`
- 验证 `BulletDef` 的命中与销毁事件

美术任务：

- 平台资源和移动反馈
- 技能特效
- 远程怪攻击演出

测试任务：

- 站平台时角色不穿透
- 高台射手能正确索敌
- AOE 半径和视觉范围一致

### 2.13.3 第三章：Boss 机制整合关

目标：

- 验证完整主线版本可玩性
- 验证多技能 Boss、结算和退出流程

配置任务：

- Boss 怪物配置
- Boss AI 组
- 多段技能与 Buff
- 关卡终点和结算触发逻辑

程序任务：

- 跑通完整战斗开始、推进、结束
- 接入胜负判断
- 接入结算界面

美术任务：

- Boss 演出
- 关键技能特效和音效
- 胜利失败反馈

测试任务：

- Boss 多阶段切换
- 死亡后重开
- 胜利后退出和再次进入

# 赛博科幻横版射击动作游戏开发流程文档

## 1. 文档目标

本文件用于把当前项目从"玩法设计提纲"细化为"可执行研发文档"，用于统一以下事项：

- 研发目标与范围
- 工程结构与模块边界
- 从需求到上线的标准开发流程
- 各系统的开发顺序、依赖关系与验收标准
- 配置、资源、代码、联调、测试、发布的协作方式

适用对象：

- 策划
- 客户端程序
- 技术美术与特效
- UI 开发
- 测试

### 1.1 配套专题文档

`Work` 目录下的专题文档用于承接本总文档中的细分规范，建议按以下方式配合使用：

- `10_设计_系统数值平衡设计.md`：核心数值、武器定位、成长与经济平衡
- `11_设计_关卡和地图设计规范.md`：关卡结构、地图生产、波次与章节节奏
- `12_设计_UI和UX交互规范.md`：模块职责、战斗 HUD、交互反馈和适配规范
- `13_设计_技术美术和特效规范.md`：动画、特效、资源挂点、表现层性能约束
- `14_设计_世界观和叙事设定.md`：世界观、势力、角色、章节叙事主题
- `17_设计_QA测试计划.md`：测试分层、专项测试、发布准入和缺陷分级

使用原则：

- 总文档负责描述研发主线、模块边界和里程碑
- 专题文档负责把某一领域展开为可执行规范
- 若专题文档与总文档冲突，以"真实工程入口 + 当前代码事实"为准，随后同步修正文档

## 2. 项目定位

### 2.1 游戏类型

- 核心类型：横版射击 + 动作闯关
- 风格定位：赛博科幻
- 操作体验：高频射击、位移闪避、技能释放、局内成长
- 目标闭环：进入关卡 -> 战斗推进 -> 击败敌人/机关 -> 完成副本 -> 结算成长 -> 进入下一关

### 2.2 当前工程基础

项目已具备以下基础框架：

- UGF 流程管理
- HybridCLR 热更入口
- Luban 配置表导出链路
- UI 模块化框架
- 战斗逻辑代理层 `GameProxyManger`
- 战斗核心模块：输入、帧同步、地图、单位、技能、时间线、伤害、AI、怪物刷新、副本
- 临时动作系统样例 `Assets/HotAssets/Scripts/GamePlayTmp/ActionSystem`

### 2.3 本文档目标产出

在当前工程基础上，逐步落地为一个可玩、可扩展、可热更迭代的横版射击动作项目，优先完成单人主线关卡闭环，再扩展更多内容。

## 4. 核心研发原则

### 4.1 逻辑与表现分离

- 逻辑层负责状态、数值、碰撞、伤害、技能、AI、关卡推进。
- 表现层负责动画、特效、音效、镜头、受击反馈、界面刷新。
- 逻辑层优先保证结果正确、可复现、可配置。
- 表现层不得反向决定逻辑结果，只能消费逻辑事件。

### 4.2 表驱动优先

以下内容优先配置化，而不是写死在代码里：

- 角色基础属性
- 怪物属性与 AI 组
- 技能、子弹、AOE、Buff、时间线事件
- 副本与关卡章节
- 怪物刷新点
- UI 模块注册信息

### 4.3 定点数约束

根据项目规范，战斗逻辑层遵守以下要求：

- 禁止使用 `float` 作为核心战斗数值
- 使用 `fix`、`fix2`、`fix3`
- 高频字符串处理使用 `ZString`
- 日志统一使用 `UnityGameFramework.Runtime.Log`
- 新增或修改函数必须补充清晰注释

### 4.4 先闭环，后扩展

研发顺序严格遵循：

1. 先跑通单角色、单地图、单关卡、单结算闭环
2. 再增加武器、技能、怪物、Boss、成长系统
3. 最后做优化、扩内容和热更发布

## 5. 从需求到上线的标准开发流程

每一个新功能、新副本、新角色都必须遵守以下流程。

### 5.1 需求确认

输入：

- 策划需求
- 参考演示
- 数值目标
- 交互说明

输出：

- 一页式需求说明
- 涉及系统清单
- 配置表增改项
- 资源需求清单
- 验收标准

准入条件：

- 需求边界明确
- 不存在"程序先做一个看看"的模糊项
- 每个需求有主负责人

### 5.2 技术拆解

程序负责人需要将需求拆成以下维度：

- 流程改动：是否涉及 Procedure、Scene、UI 跳转
- 配置改动：是否新增 Luban 表字段或新表
- 逻辑改动：涉及哪些 Proxy、Model、Unit、AI、Skill、Timeline
- 表现改动：动画、特效、音效、镜头、屏幕反馈
- 测试改动：需要哪些自测场景和回归点

输出：

- 开发任务拆解单
- 实现顺序
- 联调节点
- 风险说明

### 5.3 配置先行

优先完成配置定义，再开始写逻辑：

- 表结构确认
- Excel 配置填写
- Luban 导表
- 生成 C# 配置访问代码
- 落地测试数据

原因：

- 本项目大量系统依赖表驱动
- 先写死代码会导致后续重构成本高

### 5.4 逻辑开发

逻辑开发顺序建议如下：

1. 数据结构
2. 核心状态流转
3. 外部接口
4. 事件派发
5. 表现回调点

禁止顺序：

- 先做 UI 表现，再补底层逻辑
- 先拼动画，再补技能时间线
- 先写特例，再补公共抽象

### 5.5 表现接入

当逻辑事件稳定后，再接入：

- 动画
- 特效
- 音效
- 屏幕震动
- UI 反馈
- 伤害飘字

表现层接入要求：

- 所有效果都由明确逻辑事件驱动
- 不允许表现层自己推导伤害、命中、死亡

### 5.6 自测

开发者完成后必须做最少自测：

- 功能正确
- 空数据不崩
- 极端输入不崩
- 反复进出关卡不泄漏
- 日志中无连续报错

### 5.7 联调

联调顺序：

1. 程序与配置联调
2. 程序与动画联调
3. 程序与特效联调
4. 程序与 UI 联调
5. 关卡内容联调
6. 完整关卡回归

### 5.8 验收

验收由需求侧按本文档中的验收标准检查，验收通过后才能并入主开发分支。

### 5.9 发布

发布阶段需完成：

- 热更资源整理
- 配置版本确认
- 测试回归通过
- 发布说明与已知问题记录

## 6. 配置与资源生产流程

### 6.1 Luban 配置流程

当前项目配置链路如下：

1. 策划在 `Config/DataTables/Datas` 维护 Excel
2. 执行 `Config/gen.bat`
3. Luban 根据 `Config/DataTables/luban.conf` 导出
4. 配置代码输出到 `Assets/HotAssets/Scripts/DataTable`
5. 二进制数据输出到 `Assets/HotAssets/DataTable`
6. 运行时通过 `AppEntry.DataTable.GetDataTableLuBan<T>()` 加载

当前重点表包括：

- `TbHero`
- `TbMonster`
- `TbMainChapter`
- `TbMapData`
- `TbPlatformDef`
- `TbMonsterSpawner`
- `TbAIGroupDef`
- `TbAIActionDef`
- `TbSkillDef`
- `TbBulletDef`
- `TbSkillBulletDef`
- `TbAoeDef`
- `TbSkillAoeDef`
- `TbBuffDef`
- `TbTimeLineDef`

配置改动规则：

- 先改 Excel，再导表，再写业务逻辑
- 不直接手改自动生成的表代码
- 表字段命名保持可读性与业务语义一致
- 需要兼容旧数据时，必须写迁移说明

### 6.2 地图生产流程

地图内容生产建议如下：

1. 关卡设计搭建 Tilemap 场景
2. 配置地面阻挡层与飞行阻挡层
3. 使用 `Tools/地图/导出TileMap碰撞数据`
4. 生成碰撞字节文件到 `Assets/HotAssets/Scene/SceneAsset`
5. 填写 `MapData` 与 `PlatformDef`
6. 在 `MainChapter` 里关联 `MapId`
7. 在 `MonsterSpawner` 中配置刷怪点
8. 进入副本验证实际逻辑地图与视觉地图一致

### 6.3 动画与时间线流程

技能和动作表现建议通过时间线配置统一驱动：

1. 定义动作用途与打断规则
2. 配置 `SkillDef`
3. 关联 `TimeLineDef`
4. 在时间线节点中配置以下事件：
   - 播放动画
   - 发射子弹
   - 创建 AOE
   - 播放音效
   - 播放特效
   - 给施法者施加位移/控制状态
5. 运行时由 `TimelineProxy` 与 `SkillProxy` 消费配置

### 6.4 AI 配置流程

AI 制作流程：

1. 设计怪物行为目标
2. 拆分条件与动作
3. 配置 `AIActionDef`
4. 配置 `AIGroupDef`
5. 将 AI 组绑定到怪物或刷怪配置
6. 用测试地图验证巡逻、追击、释放技能、撤退是否符合预期

## 7. 系统开发总顺序

本项目建议按以下顺序推进，避免多个系统互相阻塞。

### 7.1 第一阶段：工程底座与可进入战斗

目标：

- 完成从启动到进入战斗场景的稳定流程
- 确保配置、UI、地图、玩家初始化链路可用

开发项：

- 校准 `GameSetting` 中流程列表
- 确认 `PreloadProcedure` 加载项完整
- 跑通 `LoginProcedure` 到 `RunGameProcedure`
- 实现战斗入口 UI
- 跑通 `FightProxy.EnterFight`
- 跑通 `MainDungeonProxy.InitDungeon`
- 跑通 `MapProxy.LoadMap`

交付物：

- 可从登录界面进入指定关卡
- 地图可加载
- 玩家可出生

验收标准：

- 首次启动无阻塞报错
- 重复进入关卡流程稳定
- 配置缺失时有明确日志

### 7.2 第二阶段：角色基础移动与输入闭环

目标：

- 完成玩家基础操作体验

开发项：

- 输入采集与命令转换
- 左右移动、跳跃、下落、朝向切换
- 地面/空中状态切换
- 受击、死亡基础切换
- 临时动作系统能力吸收到正式战斗逻辑

参考实现：

- `Assets/HotAssets/Scripts/GamePlayTmp/ActionSystem`

验收标准：

- 玩家可稳定移动、跳跃、落地
- 极端输入下状态不乱跳
- 动作切换没有明显卡死状态

### 7.3 第三阶段：射击、技能、伤害闭环

目标：

- 完成横版射击核心战斗体验

开发项：

- 基础普通攻击
- 子弹创建、飞行、命中、销毁
- 技能释放与时间线驱动
- AOE 创建与结算
- Buff 生效与移除
- 伤害结算与死亡处理

核心模块：

- `SkillProxy`
- `TimelineProxy`
- `BulletModel`
- `DamageProxy`
- `BuffModel`
- `EffectProxy`

验收标准：

- 普攻、技能、AOE、Buff 均可由配置驱动
- 命中、伤害、死亡结果稳定
- 表现层能正确消费命中和释放事件

### 7.4 第四阶段：敌人、刷怪与 AI

目标：

- 形成基础闯关对抗

开发项：

- 怪物实体初始化
- 刷怪器配置驱动
- AI 巡逻、跟随、攻击、撤退
- 怪物技能释放
- 关卡波次推进

核心模块：

- `MonsterSpawnerProxy`
- `AIProxy`
- `UnitProxy`
- `RoleMonster`

验收标准：

- 至少 3 类敌人可正常工作
- 刷怪波次能按条件推进
- AI 在边界情况下不会原地卡死或无限抖动

### 7.5 第五阶段：副本流程与胜负结算

目标：

- 跑通完整关卡闭环

开发项：

- 关卡开始、推进、结束状态
- 胜利条件、失败条件
- 章节选择与进入
- 战斗结算界面
- 返回主界面或下一关

核心模块：

- `FightProxy`
- `MainDungeonProxy`
- `RunGameProcedure`
- `FightMainChapter` UI 模块

验收标准：

- 主线关卡从选择到结束完整可玩
- 玩家死亡和胜利都能进入正确结算
- 退出关卡后状态能清理干净

### 7.6 第六阶段：HUD、背包、成长与 Meta 系统

目标：

- 完成战斗外成长与战斗中反馈

开发项：

- 战斗 HUD
- 血条、能量、弹药、技能冷却、伤害飘字
- 背包与掉落物
- 道具与奖励结算
- 角色成长、技能升级、属性强化

当前 UI 基础目录：

- `Assets/HotAssets/Scripts/UI/Core`
- `Assets/HotAssets/Scripts/UI/Module/FightShare`
- `Assets/HotAssets/Scripts/UI/Module/FightMainChapter`

验收标准：

- HUD 信息与逻辑数据同步
- 升级、掉落、结算链路可回放验证
- UI 反复打开关闭无明显状态残留

### 7.7 第七阶段：表现强化与打磨

目标：

- 提升手感与观感

开发项：

- 枪口火焰、命中特效、死亡特效
- 受击闪白、屏幕震动、顿帧
- 武器差异化表现
- 不同怪物受击反馈
- Boss 演出

验收标准：

- 不改变逻辑结果前提下提升反馈强度
- 高压战斗场景仍保持稳定帧率

### 7.8 第八阶段：优化、热更与发布

目标：

- 达到可持续迭代状态

开发项：

- 性能分析
- GC 清理
- 配置热更新流程确认
- 资源依赖与引用计数检查
- 崩溃点与空引用清理
- 发布 checklist 固化

验收标准：

- 主流程可稳定热更迭代
- 常玩路径无严重卡顿和内存异常
- 测试版本具备发布说明和已知问题列表

## 8. 详细模块开发说明

### 8.1 Procedure 流程层

职责：

- 控制启动、预加载、场景切换、登录、运行中状态

开发要求：

- 所有场景切换通过流程层或事件统一控制
- 不在业务脚本中直接写死跨场景初始化逻辑

重点文件：

- `Assets/HotAssets/Scripts/Procedures/PreloadProcedure.cs`
- `Assets/HotAssets/Scripts/Procedures/ChangeSceneProcedure.cs`
- `Assets/HotAssets/Scripts/Procedures/LoginProcedure.cs`
- `Assets/HotAssets/Scripts/Procedures/RunGameProcedure.cs`

### 8.2 GamePlay 逻辑层

职责：

- 管理战斗内所有核心逻辑

建议分工：

- `Frame`：帧推进、本地模式/房间模式兼容
- `GameInput`：输入对象采集与回放
- `Map`：逻辑地图、碰撞、平台数据
- `Unit`：角色、子弹、AOE 的统一对象层
- `Player`：玩家控制
- `Skill`：技能释放入口
- `TimeLine`：技能事件调度
- `Bullet`：子弹行为
- `Damage`：伤害结算
- `Buff`：状态与属性修改
- `AI`：怪物决策
- `MonsterSpawner`：刷怪控制
- `Dungeon`：关卡推进

### 8.3 UI 层

职责：

- 登录
- 主章节选择
- 战斗 HUD
- 浮层信息
- 结算与成长界面

开发要求：

- UI 只读取模型数据或订阅事件
- 不直接处理战斗规则
- 模块跳转由 `UIModuleManager` 控制

### 8.4 数据表层

职责：

- 为所有战斗与界面模块提供稳定数据输入

要求：

- 表字段一经发布，修改必须评估兼容性
- 配置值缺失时必须有容错日志
- 表与逻辑之间通过明确访问层，不直接散落读取

## 9. 单个功能的标准落地模板

以后新增一个功能时，按以下模板执行。

### 9.1 以"新增一把电磁步枪"为例

1. 策划定义需求：射速、伤害、后坐力、子弹类型、音效、特效
2. 程序确认是沿用现有 `Skill/Bullet/Timeline` 体系，还是单独新增武器表
3. 配置填写测试数据
4. 导表并校验读取
5. 实现或复用子弹策略与命中逻辑
6. 接入动画、音效、特效
7. 接入 HUD 弹药显示
8. 在测试关卡中验证 DPS、手感、命中稳定性
9. 修正平衡数值
10. 通过验收后合入主线

### 9.2 以"新增一个远程怪"为例

1. 配置怪物基础属性
2. 配置 AI 行为组
3. 配置技能与子弹
4. 放入 `MonsterSpawner`
5. 接入动画与受击表现
6. 验证追击、索敌、射击、死亡
7. 验证和玩家技能交互是否正常

## 11. 版本管理与协作建议

### 11.1 任务切分原则

- 一个功能只对应一个明确责任人
- 公共底层改动优先于内容层开发
- 同时涉及配置、代码、资源时，先锁定配置结构

### 11.2 提交要求

- 提交前必须自测
- 提交说明必须说明改动范围
- 涉及配置表改动时必须说明对应表和字段
- 涉及联动模块时必须写明影响面

### 11.3 禁止事项

- 直接修改自动生成代码后不保留生成来源
- 在逻辑层混入大量表现控制
- 未定义验收标准就进入开发
- 不经过测试直接合入主流程

## 12.1 按章节的执行拆解

为了让里程碑不是空标题，首版建议直接按 3 个章节拆为可执行任务包。

### 第一章任务包：基础闭环章

目标：

- 从登录到主线第一关完整跑通
- 角色基础操作、普通攻击和基础怪物成立

配置任务：

- `MainChapter` 新建 1 条
- `MapData` 新建 1 条
- `Monster` 至少 2 条
- `MonsterSpawner` 至少 2 条
- `SkillDef / TimeLineDef / SkillBulletDef / BulletDef` 各至少 1 条

程序任务：

- 跑通 `FightMainChapter -> FightProxy -> MainDungeonProxy -> MapProxy`
- 跑通 `UnitProxy.InitRole`
- 跑通基础子弹命中和怪物死亡

验收：

- 可稳定进入第一关
- 玩家可移动、跳跃、普攻
- 基础怪可触发、受击、死亡

### 第二章任务包：技能与空间章

目标：

- 建立高低差、平台、远程火力和技能拆场体验

配置任务：

- `PlatformDef` 至少 1 组
- `Monster` 追加远程怪
- `AIGroupDef / AIActionDef` 追加远程逻辑
- `SkillAoeDef / AoeDef / BuffDef` 至少 1 套

程序任务：

- 验证平台逻辑稳定
- 验证 `Timeline` 节点驱动技能
- 验证 AOE 与 Buff 生效

验收：

- 平台可站立、可携带角色移动
- 远程怪能追踪或压制
- 玩家技能可明显改变战斗节奏

### 第三章任务包：Boss 与结算章

目标：

- 形成一个能对外演示的完整主线版本

配置任务：

- Boss 怪物
- Boss AI
- Boss 技能链
- 最终关卡入口和结算内容

程序任务：

- 胜负条件
- 结算界面
- 退出和重进

验收：

- 至少 1 个 Boss 可完整打完
- 胜利失败都可进入正确结算
- 退出后状态可清理并再次挑战

## 13. 最终验收定义

达到以下条件，可认为项目完成第一阶段正式可玩版本：

- 从启动到进入主线关卡流程稳定
- 至少 1 名玩家角色完整可用
- 至少 3 类普通怪 + 1 个精英或 Boss 可用
- 至少 1 个完整主线关卡可从进入打到结算
- 技能、子弹、AOE、Buff、AI、刷怪均可配置驱动
- HUD、结算、章节入口可正常工作
- 主流程无阻塞性 Bug
- 可支持后续通过配置与热更继续扩展内容

## 14. 开发执行结论

本项目当前最合理的执行顺序不是继续扩写玩法概念，而是按现有工程基础，优先完成以下主线：

1. 流程层稳定
2. 战斗基础闭环
3. 怪物与副本闭环
4. UI 与成长闭环
5. 表现强化
6. 优化与发布

后续所有需求都应挂靠到本文档中的某个阶段、某个模块和某个验收标准，避免研发过程继续发散。

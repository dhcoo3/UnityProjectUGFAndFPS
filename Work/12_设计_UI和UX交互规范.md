# 3. UI/UX 交互说明文档 (UI/UX Spec)

## 3.1 文档目标

本文档用于统一项目的界面结构、交互反馈、信息优先级和适配规范，确保以下内容协同一致：

- `UIModuleManager` 的模块注册与跳转
- 登录、主章节、战斗 HUD、结算等模块设计
- UI 程序实现、UI 美术出图和动效接入
- UI 测试验收和回归标准

本项目 UI 的核心目标不是“堆信息”，而是让玩家在高速战斗中第一时间读到最关键的状态。

## 3.2 设计原则

- 战斗信息优先于装饰信息
- 瞬时反馈优先于长期展示
- 可读性优先于赛博装饰密度
- UI 只展示和消费状态，不直接决定战斗逻辑
- 同一类交互在所有模块中保持一致反馈

## 3.3 当前模块结构

根据现有工程，UI 主要围绕以下模块展开：

- `Login`
- `FightMainChapter`
- `FightShare`
- `Core/UIModuleManager`

建议职责如下：

- `Login`：登录、入口、初始引导
- `FightMainChapter`：章节选择、关卡入口、进入战斗
- `FightShare`：战斗 HUD、伤害飘字、角色状态展示
- `Result`：结算、奖励、再来一次、返回主界面

## 3.4 核心流程设计

### 3.4.1 启动到战斗入口

建议 UI 流程如下：

1. 预加载页显示加载进度
2. 进入登录界面
3. 登录或单机确认后进入章节入口
4. 选择章节与关卡
5. 点击开始战斗
6. 切入战斗加载态
7. 进入战斗 HUD

要求：

- 任何阶段都要有明确状态，不允许“点了没反应”
- 切场景与开战之间要能显示加载进度或过渡文案
- 重复点击开始按钮时，必须有去重处理或禁用态

### 3.4.2 战斗结束流程

建议流程：

1. 战斗结束冻结输入
2. 播放胜利或失败反馈
3. 打开结算面板
4. 展示评级、奖励、统计信息
5. 提供返回章节、再次挑战、继续下一关入口

## 3.5 战斗 HUD 规范

### 3.5.1 信息优先级

战斗 HUD 的信息优先级建议如下：

第一优先级：

- 当前生命值
- 当前弹药
- 技能冷却
- 危险状态提示

第二优先级：

- 目标血条
- Buff / Debuff
- 掉落与拾取提示

第三优先级：

- 连击、评分、统计信息
- 非关键演出提示

### 3.5.2 HUD 布局建议

- 左上：玩家生命、护盾、能量
- 右下：武器图标、弹匣、换弹状态、主要技能按钮
- 中下：交互提示、拾取提示、特殊机制提示
- 角色头顶或目标附近：伤害飘字、异常状态、锁定标记
- 顶部中间：Boss 血条或波次信息

### 3.5.3 战斗中状态反馈

必须明确显示的状态：

- 换弹中
- 技能冷却中
- 受击或濒死
- 异常状态生效中
- 关键机制倒计时

建议表现：

- 换弹时武器区半透明并显示进度环
- 技能冷却用灰遮罩加数字倒计时
- 濒死状态加入边框脉冲或屏幕边缘提示
- 关键 Debuff 使用高对比图标与剩余时间

## 3.6 交互反馈规范

### 3.6.1 通用按钮反馈

所有按钮至少具备以下反馈：

- 按下缩放到 `0.95`
- 抬起恢复到 `1.0`
- Hover 或聚焦态有高亮描边或发光
- Disabled 态透明度降低且不可点击

### 3.6.2 错误反馈

当玩家操作未生效时，必须有明确反馈：

- 模块未开启：提示未解锁原因
- 战斗中不能打开：提示当前状态不可操作
- 网络或资源未准备好：提示等待或重试

禁止：

- 仅打印日志，不给 UI 提示
- 只做按钮抖动，不说明原因
- 使用过长弹窗阻断高频操作

### 3.6.3 伤害飘字

建议统一规则：

- 敌人普通伤害：白色
- 敌人暴击伤害：黄色
- 玩家受伤：红色
- 治疗或回复：绿色
- 护盾吸收：青色

运动规则：

- 默认 `0.5s - 0.8s` 内向上漂浮并淡出
- 暴击数字略大并增加轻微放大动画
- 同一帧多次伤害应合并或错位，避免完全重叠

## 3.7 章节选择与关卡入口

`FightMainChapter` 模块建议承担以下信息展示：

- 当前章节名称
- 已解锁关卡
- 推荐战力或推荐等级
- 关卡奖励预览
- 关卡目标标签

交互要求：

- 未解锁关卡显示锁定态与解锁条件
- 已通关关卡显示评级或通关标记
- 点击开始后，章节入口需立即进入“处理中”状态

## 3.8 结算界面规范

结算界面至少展示以下信息：

- 通关结果
- 完成时间
- 击败敌人数
- 受到伤害次数或总量
- 奖励列表
- 下一步操作按钮

设计原则：

- 胜利页突出正反馈和奖励获取
- 失败页突出失败原因和重试动机
- 结算页停留信息量适中，不做冗长演出阻塞操作

## 3.9 视觉风格与字体

### 3.9.1 色彩规范

建议主色如下：

- 主高亮：荧光蓝 `#00F0FF`
- 警告色：赛博黄 `#F0E68C`
- 危险色：故障红 `#FF4D6D`
- 特殊强调：霓虹紫 `#FF00FF`
- 背景基色：深蓝灰 `#101826`

使用规则：

- 不同功能状态用颜色区分，不同模块保持一致
- 警告色仅用于高优先级提醒，避免满屏黄字
- 危险色只用于受伤、失败、紧急状态

### 3.9.2 字体规范

- 标题字体可使用科技感较强字体
- 正文字体优先保证中英文可读性
- 数值字体建议使用等宽或半等宽字体，便于快速识读

## 3.10 分辨率与适配

- 参考分辨率：`1920x1080`
- 最低适配目标：`1280x720`
- 默认宽高比优先支持 `16:9`

适配规则：

- 核心 HUD 不依赖背景图尺寸，使用锚点和安全边距布局
- 关键按钮不得贴边过近，避免异形分辨率误触
- 大面板优先居中，边缘装饰允许裁切

## 3.11 程序实现约束

结合当前 `Core` 模块结构，UI 程序应遵循：

- `Controller` 处理事件和模块跳转
- `Model` 负责数据组织
- `View` 负责表现和组件绑定
- 业务跳转统一走 `UIModuleManager`

禁止：

- 直接在 View 中发起复杂业务逻辑
- UI 层直接计算战斗伤害或规则
- 模块间通过硬编码对象引用互相调用

## 3.12 UI 验收标准

每个模块至少通过以下检查：

- 能正常打开和关闭
- 重复打开关闭不报错、不残留旧状态
- 关键按钮点击后有明确反馈
- 关键数值与逻辑状态同步
- 在目标分辨率下无明显错位和遮挡
- 美术资源缺失时有降级或错误提示

## 3.13 当前 UI 模块落地对照

当前 `ModuleType` 枚举包括：

| 枚举 | 含义 |
| :--- | :--- |
| `Home = 1` | 主页 |
| `FightCommon = 2` | 战斗通用 |
| `FightMainChapter = 3` | 主战斗章节 |
| `Login = 4` | 登录 |

当前 `GameModule` 表字段仅包括：

| 字段 | 含义 |
| :--- | :--- |
| `Id` | 模块枚举 |
| `Entry` | 入口面板路径 |

这意味着当前 UI 模块配置更偏“入口注册”，还不包含：

- 开启条件
- 层级
- 返回栈策略
- 预加载标记
- 模块依赖关系

如果后续要把 UI 完整配置化，建议扩 `GameModule` 表。

## 3.14 按阶段的 UI 任务拆解

### 3.14.1 登录阶段

必须完成：

- 登录面板打开
- 按钮可响应
- 进入章节界面

建议补充：

- 当前版本号显示
- 资源加载提示
- 异常状态提示

### 3.14.2 章节入口阶段

必须完成：

- 关卡列表显示
- 点击开始进入战斗
- 禁止重复点击

建议补充：

- 关卡名称
- 推荐难度
- 奖励预览
- 通关标记

### 3.14.3 战斗 HUD 阶段

必须完成：

- 生命值
- 弹药或资源
- 技能冷却
- 伤害飘字

建议补充：

- Buff 图标
- Boss 血条
- 波次提示
- 危险提示

### 3.14.4 结算阶段

必须完成：

- 胜利失败
- 奖励展示
- 返回入口

建议补充：

- 评分
- 用时
- 击杀统计
- 再来一次

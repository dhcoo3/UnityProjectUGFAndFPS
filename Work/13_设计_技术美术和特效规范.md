# 4. 表现层与美术技术规范 (Technical Art Spec)

## 4.1 文档目标

本文档用于统一战斗表现层、角色动画、特效、地图美术资源和性能约束，保证技术美术、程序和特效协作时遵守同一套落地规则。

本文档重点解决以下问题：

- 逻辑层与表现层边界不清
- 动画事件与技能时间线重复驱动
- 特效资源失控、池化规则不统一
- 地图分层与碰撞信息混杂
- 性能预算缺失导致战斗场景抖动

## 4.2 表现层总体原则

- 表现永远消费逻辑事件，不反向改写逻辑结果
- 表现层允许插值和平滑，但不得篡改逻辑位置
- 一个效果只允许一个权威驱动源，避免重复触发
- 所有高频对象优先池化，不走频繁创建销毁
- 视觉反馈必须服务战斗可读性，而不是单纯堆发光和粒子

## 4.3 逻辑与表现解耦规范

### 4.3.1 事件驱动原则

建议表现层只订阅明确业务事件，例如：

- `OnUnitSpawn`
- `OnUnitDead`
- `OnUnitHit`
- `OnSkillCast`
- `OnBulletCreate`
- `OnAoeCreate`
- `OnSceneLoadProgress`

要求：

- 动画、音效、镜头、屏幕震动都应挂靠到事件
- 表现层不得根据 Animator 状态猜测技能是否命中
- 表现层不得通过粒子碰撞来补做逻辑伤害

### 4.3.2 插值与同步

当前工程存在 Logic 和 Render 分层，表现同步建议如下：

- Logic 层输出权威位置和状态
- Render 层每帧从 Logic 读取 `fix` 坐标
- Render 层可使用 `Lerp` 或自定义插值做平滑显示
- 受击、冲刺、强制位移等高识别动作应优先保证响应感，不宜过度平滑

## 4.4 角色动画规范

### 4.4.1 状态命名

角色动画状态机至少应覆盖以下命名：

- `Idle`
- `Run`
- `Jump`
- `Fall`
- `Dash`
- `Hit`
- `Dead`
- `Attack_Start`
- `Attack_Active`
- `Attack_End`
- `Skill_Start`
- `Skill_Active`
- `Skill_End`

规则：

- 角色通用状态尽量统一命名，减少程序适配成本
- 攻击与技能动作建议拆成前摇、生效、后摇三个阶段
- Boss 可增加专属状态，但通用入口仍应保留统一触发方式

### 4.4.2 动画事件规则

动画事件只负责“通知表现节点”，不负责决定战斗结果。

允许的事件类型：

- `OnEffectSpawn`
- `OnFootstep`
- `OnShellDrop`
- `OnWeaponShake`

谨慎使用的事件：

- `OnAttackHit`

说明：

- 若命中时机已由 `Timeline` 驱动，则动画事件不再重复做命中
- 当技能以 `TbTimeLineDef` 为准时，动画事件只补视觉，不补规则

## 4.5 特效规范

### 4.5.1 特效分类

建议统一分为：

- `OneShot`：一次性播放后回收
- `Loop`：持续挂载在单位或场景上
- `Attach`：绑定骨骼或挂点
- `Scene`：场景机关、环境或氛围特效

### 4.5.2 命名规范

命名建议：

- `VFX_Hero_Weapon_Muzzle_01`
- `VFX_Monster_Hit_Spark_01`
- `VFX_Scene_LaserLoop_01`

要求：

- 命名中带上归属对象和用途
- 不使用 `New Effect`、`Test01` 这类无意义命名

### 4.5.3 性能预算

首版参考预算：

- 单个普通子弹命中特效粒子数尽量不超过 `20 - 40`
- 普通战斗同屏实时 Loop 特效不超过 `10`
- Boss 战允许更高预算，但必须有 LOD 或阶段性关闭策略

优化原则：

- 高复用特效必须走对象池
- 同屏高频命中特效允许用简化版替代
- 透明叠加面积过大时优先压缩时长和层数

## 4.6 音效与镜头反馈

### 4.6.1 音效规则

- 枪声优先保证辨识度和节奏感
- 受击音效要能区分玩家受击、敌人受击和护盾吸收
- 大招、Boss 蓄力、危险机关必须有预警音

### 4.6.2 镜头反馈规则

- 普通命中使用轻微镜头抖动或 Hit Stop
- 强力技能可使用中强度震动
- Boss 大招可结合镜头拉远、屏幕边缘提示

禁止：

- 所有攻击都套同一强度震动
- 高压战斗里持续晃屏，导致信息不可读

## 4.7 地图美术资源规范

### 4.7.1 Tile 与图集

- Tile 尺寸：`1x1 Unity Unit`
- 同章节主地图元素尽量使用同一图集
- 碰撞层素材与视觉层素材分开维护

### 4.7.2 层级建议

- Layer 0：远景背景
- Layer 1：中景装饰
- Layer 2：可玩区域主地形
- Layer 3：前景遮挡
- Layer 4：特效与动态装饰

要求：

- 美术层级必须与逻辑碰撞层分离
- 前景遮挡不应长期遮挡玩家和关键敌人
- 高对比元素不要放在玩家常驻活动路径中心

## 4.8 角色资源与挂点规范

角色 Prefab 建议统一保留以下挂点：

- `Root`
- `Body`
- `Weapon`
- `Muzzle`
- `HitPoint`
- `Head`
- `Foot_L / Foot_R`

作用：

- 统一枪口火焰、受击火花、脚步尘土、头顶 UI 的接入位置
- 降低不同角色间的接入差异

## 4.9 资源目录与交付规范

建议资源交付时至少包含：

- Prefab
- 动画片段
- Animator Controller
- 材质与贴图
- 特效预设
- 音效配置说明
- 使用说明和挂点说明

交付前检查：

- 名称规范
- 贴图尺寸和压缩格式合理
- 无丢失引用
- 可被对象池复用

## 4.10 技术美术联调清单

每次新角色或新技能接入时，至少检查：

- 逻辑事件与动画播放时序是否一致
- 枪口、子弹、命中特效位置是否对齐
- 死亡和受击反馈是否能正确回收对象
- 同屏高压下是否出现明显掉帧
- 切场景后是否有残留特效和音效

## 4.11 验收标准

表现层内容通过验收需满足：

- 不改变逻辑结果
- 关键动作可读
- 同屏高压仍能保持稳定
- 资源命名和目录规范清晰
- 无明显闪烁、穿帮、挂点错位和回收泄漏

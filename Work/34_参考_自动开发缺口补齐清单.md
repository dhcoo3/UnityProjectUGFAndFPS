# 自动开发缺口补齐清单

本文档基于当前 `Work` 目录内容，整理自动开发前建议补齐的缺口。

结论：
- 主线三章闭环已经基本成型。
- 当前最缺的是“口径统一”和“收口文档”。
- 其次才是存档、设置、扩展内容等外围系统。

---

## 1. P0 必补

### 1.1 版本口径统一

问题：
- `00_约束_项目总体游戏设计文档.md` 仍然保留完整工程视角。
- `01_约束_AI开发交接说明.md` 已明确当前主线是 `3` 章闭环。
- `31_参考_12章开发快速指南.md` 仍是 `12` 章扩展版口径。
- `30_参考_完整文档导航索引.md` 中存在旧命名引用，容易误导自动开发。

建议补齐：
- 增加一份“当前版本范围说明”。
- 明确本轮只做三章闭环，12 章只作为后续扩展参考。
- 把旧文档中的过时引用统一标记为“扩展版 / 历史参考”。

### 1.2 战斗结算闭环

问题：
- 退出、失败、重新挑战、返回主菜单的时序还没有一份独立收口文档。
- `GameOverProcedure`、`FightProxy.EndFight`、`FightProxy.ExitFight` 的职责边界需要固定。

建议补齐：
- 一份“胜负结算与退出流程说明”。
- 明确胜利、失败、重新进入、回主菜单的完整链路。
- 补一张时序图或步骤表。

### 1.3 配置校验与容错规则

问题：
- 目前文档里有配置表说明，但缺少统一的“导表后检查规则”。
- 自动开发最怕读到缺字段、错 ID、错枚举却继续写逻辑。

建议补齐：
- 新增“配置校验清单”。
- 统一规定：缺字段、外键不存在、类型错误时如何报警和降级。
- 明确哪些错误必须阻断，哪些可以继续跑。

## 2. P1 应补

### 2.1 UI 与模块收口

建议补：
- `UI` 模块总表，列出当前已注册模块、入口、依赖、打开条件。
- `HUD / 结算 / 章节入口 / 登录` 的责任边界说明。
- 重复打开、重复订阅、关闭回收的检查规则。

### 2.2 地图与资源命名规范

建议补：
- 地图场景名、碰撞导出名、MapData.Name 的统一命名规则。
- 资源路径与配置字段的对应关系。
- 失败案例示例，避免自动生成错名。

### 2.3 开发自测模板

建议补：
- 每做完一个功能，固定返回什么内容。
- 最小自测清单模板。
- 日志检查模板。

### 2.4 运行时故障排查

建议补：
- 常见报错分类。
- 场景没进、怪不刷、技能不放、结算卡住的排查路径。
- “先查配置，再查流程，再查表现”的固定顺序。

### 2.5 章节级任务索引

建议补：
- 一份“第一章 / 第二章 / 第三章”的任务状态表。
- 每章已完成、未完成、阻塞项、验收项分开列。

## 4. 推荐新增文档

优先级最高的新增文档建议是：

1. `35_约束_当前版本范围说明.md`
2. `36_开发_结算与退出流程说明.md`
3. `37_开发_配置校验与容错规范.md`

## 5. 建议执行顺序

1. 先统一版本口径。
2. 再补结算与退出闭环。
3. 再补配置校验和自测模板。
4. 最后再做 UI、存档、设置、扩展内容。

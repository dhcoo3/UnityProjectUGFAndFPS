# UI 模块与入口总表

本文档用于汇总当前工程里已知的 UI 模块、入口、依赖和落点，方便自动开发直接查找。

## 1. 当前目标

- 快速定位 UI 模块职责。
- 明确 UI 跳转入口。
- 明确哪些模块已知、哪些模块只是预留。
- 避免新增 UI 时重复造入口。

## 2. 当前已知模块

### 2.1 `Login`

定位：
- 登录入口
- 单机或联网确认
- 初始引导

相关入口：
- `LoginProcedure`
- `LoginController`
- `LoginPanel`

依赖：
- `UIModuleManager`
- `GameModule`
- 登录流程状态

当前职责：
- 打开登录界面
- 处理开始按钮
- 进入后续战斗或章节入口

### 2.2 `FightMainChapter`

定位：
- 章节入口
- 关卡选择
- 战斗开始前的主界面

相关入口：
- `FightMainChapterController`
- `FightMainChapterPanel`

依赖：
- 章节数据
- `MainChapter`
- `MapData`
- 关卡解锁状态

当前职责：
- 展示章节信息
- 点击开始进入战斗
- 处理重复点击去重

### 2.3 `FightShare`

定位：
- 战斗通用 UI
- HUD
- 伤害飘字
- 角色状态展示

相关入口：
- `FightShareController`
- `FightShareView`
- `OverlayFightHudUI`
- `OverlayFightDamageUI`
- `RoleHud`

依赖：
- 战斗中角色数据
- 伤害事件
- 技能冷却
- 状态事件

当前职责：
- 显示战斗中的核心状态
- 订阅战斗事件并刷新
- 加载共享战斗 prefab

### 2.4 `GameOverProcedure` 对应结算 UI

定位：
- 胜利结算
- 失败结算
- 返回主菜单
- 重新挑战

相关入口：
- `GameOverProcedure`
- `GameOverUIForm`

说明：
- 当前更偏流程层收口。
- 具体 UI 名称以工程实际为准。

## 3. 当前模块注册口径

当前文档里已知的注册动作：

- 读取 `TbGameModule`
- 注册 `Controller / Model / View`
- 打开模块入口
- 检查模块是否已打开

当前已知 `ModuleType`：

- `Home`
- `FightCommon`
- `FightMainChapter`
- `Login`

## 4. 当前 `GameModule` 表状态

当前 `GameModule` 表字段口径偏轻量，主要包括：

- `Id`
- `Entry`

这意味着当前更接近“入口注册表”，还不是完整 UI 配置表。

建议后续再补的配置项：

- 开启条件
- 层级
- 返回栈策略
- 预加载标记
- 模块依赖关系

## 5. 入口映射建议

| 模块 | 主要入口 | 典型内容 |
| :--- | :--- | :--- |
| `Login` | `LoginProcedure` | 登录、开始、引导 |
| `FightMainChapter` | `FightMainChapterController` | 章节入口、关卡选择 |
| `FightShare` | `FightShareController` | HUD、飘字、角色状态 |
| `Result/GameOver` | `GameOverProcedure` | 结算、返回、重试 |

## 6. 开发规则

- UI 只负责显示和交互，不直接决定战斗结果。
- 新模块必须同步补 `TbGameModule` 和 `UIModuleManager`。
- Prefab、脚本、配置的名称要一致。
- 重复打开和关闭必须可用。
- 战斗中 UI 不能反向污染结算或逻辑状态。

## 7. 最小验收

- 能找到对应入口。
- 能正常打开和关闭。
- 重复打开不残留状态。
- 关键按钮有反馈。
- 数据变更后能刷新 UI。
